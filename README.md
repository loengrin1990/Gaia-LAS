# Local Analytics System MVP

Локальная аналитическая оболочка для работы с проектной памятью Obsidian,
маскированием, транскрибацией, LM Studio и безопасной подготовкой запросов для
Codex/ChatGPT.

## Запуск

```bash
python3 app.py
```

Затем открыть:

```text
http://127.0.0.1:8787
```

HTML и JSON-ответы Gaia отдаются с `Cache-Control: no-store`, чтобы браузер не
держал старую версию интерфейса после перезапуска сервиса. В шапке UI есть
чип `Gaia UI: 2026-07-04`; если его не видно, открыта старая вкладка или
другая копия Gaia.

Или запустить файл в корне workspace:

```text
Запустить аналитическую систему.command
```

Перед запуском Gaia читает и проверяет:

```text
Local_Analytics_System/config.json
```

Для нового окружения можно начать с `config.example.json`, скопировать его в
`config.json` и затем указать локальные пути. `config.json` не предназначен для
публикации в git.

В конфиге `config_version = 1` задает текущий контракт настроек. Старые локальные
`config.json` без этого поля читаются как версия 1; неподдержанная версия
останавливает запуск с понятной ошибкой.

В конфиге задаются пути Obsidian, сервисной документации, рабочей папки,
транскрибатора, launchers, endpoint LM Studio, host/port, локальные LLM-лимиты,
retention Archive и опциональный semantic rerank Lore через локальную LM Studio.

Секция `local_llm` управляет безопасной отправкой prompt в LM Studio:

```json
"local_llm": {
  "prompt_char_limit": 16000,
  "max_tokens": 1200
}
```

Для моделей уровня `google/gemma-4-26b-a4b-qat` рекомендуется сначала выставить
в LM Studio `Context Length = 16384`, затем оставить `prompt_char_limit = 16000`
в Gaia. Если LM Studio загружена с меньшим окном, например `8192`, лимит Gaia
нужно снизить до `8000-12000`. Ошибка LM Studio вида `tokens to keep ... greater
than the context length` означает, что фактическое окно загруженной модели меньше
отправленного локального prompt; в этом случае уменьши `prompt_char_limit` или
перезагрузи модель с большим `Context Length`.

## Принципы MVP

- ПД не отправляются за пределы локальной машины.
- Внешний анализ через Codex/ChatGPT возможен только после маскирования и ручного
  подтверждения пользователем.
- Если маскирование невозможно, система предлагает локальный маршрут через LM Studio.
- Obsidian остается базой знаний и журналом работы.
- Документация сервиса хранится вне `Проекты`, чтобы не попадать в выбор
  аналитического контекста Gaia.
- Проекты могут входить в группы; групповая память хранит надпроектные
  регламенты, шаблоны и методики, которые наследуются проектами.
- Интерфейс только координирует модули; тяжелые операции могут выполняться в
  отдельных локальных окнах.
- Имена подсистем `Scribe`, `Lore`, `Veil` остаются в интерфейсе как устойчивые
  метки, но рабочие кнопки называются по бизнес-действию: собрать контекст,
  проверить данные, предложить записи, записать в память.

## Модули

- Gaia - оркестратор запроса.
- Focus - браузерный рабочий интерфейс: `Диалог`, `Память`, `Проекты`,
  `Диагностика`.
- Veil - локальное маскирование ПД.
- Echo - локальная транскрибация аудио/видео.
- Lore - чтение эффективного контекста Obsidian: группа + проектная память + точечный evidence drill-down в первичные источники.
- Project Registry - создание, редактирование, группировка и проверка
  проектных пространств памяти.
- Project Dialogues - история диалогов по каждому проекту и продолжение turn с rolling summary.
- Scribe - Inbox для файлов проекта и staged обновление памяти внутри Gaia после review.
- Forge - сборка безопасного аналитического пакета.
- Hearth - локальный fallback через LM Studio.
- Archive - журналирование запросов и результатов в Obsidian.

Кодовая структура сейчас намеренно плоская: `gaia/orchestrator.py` собирает
pipeline, `gaia/models.py` держит публичные dataclass-контракты, а модули
`masking.py`, `memory.py`, `scribe.py`, `projects.py`, `archive.py` и другие
закрывают доменные роли без дробления на подпакеты.

HTTP-слой использует только стандартную библиотеку Python: multipart-запросы
разбираются через `email.parser`, JSON и multipart body имеют явные лимиты, а
асинхронные задачи `/api/analyze` выполняются в bounded executor
`MAX_WORKERS = 4`.

Внешние Obsidian helpers (`privacy_masker.py`, `update_project_sources.py`)
подключаются лениво во время фактического вызова. Импорт `gaia.masking` и
`gaia.extraction` сам по себе не меняет глобальный `sys.path`.

## Проверка config

```bash
python3 -B -m gaia.config
```

Команда должна вывести `Gaia config OK` и рабочий URL. Если путь неверный или
порт задан некорректно, Gaia остановится с понятной ошибкой до запуска сервера.

## Обновление памяти

После готовой задачи обработки или сообщения диалога в Focus доступна верхняя
вкладка `Память`. Она показывает новые файлы проекта, разбирает выбранный файл,
предлагает карточки для памяти и записывает только выбранные карточки после
подтверждения пользователя. Технически это по-прежнему endpoints Scribe:
`POST /api/scribe-plan` готовит предложения, а `POST /api/scribe-apply`
записывает выбранные `selected_item_ids`.

Список файлов во вкладке `Память` показывает файлы выбранного проекта, которые
еще требуют обработки. Уже записанные в память источники скрываются из списка,
хотя физически остаются в `Исходники`. `GET /api/scribe-inbox` возвращает
кандидаты, `GET /api/scribe-inbox/preview` показывает предварительный просмотр,
а кнопка `Разобрать выбранный файл` запускает обычный pipeline обработки для
выбранного файла.

Список файлов - это не список того, что автоматически попадет в память. Gaia
читает только выбранный файл после кнопки `Разобрать выбранный файл`.
`Память_Graph`, служебные memory-файлы, `Автоизвлеченный текст`, скрытые файлы
и неподдерживаемые расширения не попадают в Inbox. После успешной записи
выбранного файла в память он получает статус `indexed` и исчезает из списка.
Предложения в память для Inbox ограничены выбранным файлом и не подмешивают
`source_summary` из других Lore/evidence источников проекта.

Для `.xlsx` Gaia строит структурный предварительный просмотр на стандартной библиотеке Python:
листы, заголовки, первые строки, merged ranges и formula cells. В Scribe
попадает markdown-нормализация таблицы, а не сырой workbook. Это снижает риск
переноса табличного шума в память.

Перед записью Gaia создает backup активной памяти в service docs, затем создает
новые graph nodes и append-only записи в `<код> - Источники.md`,
`<код> - Журнал памяти.md` и graph index. Markdown-черновик через
`POST /api/scribe-draft` сохранен как export-режим без записи памяти.

## Профили анализа

Focus загружает профили через `GET /api/profiles`, а `POST /api/analyze`
передает выбранный `profile`. Forge использует профиль как prompt template:
общий анализ, записка для решения, риск-анализ, кандидаты в память или план
действий. Если профиль не передан, используется `general`. Под select профиля
Focus показывает короткую подсказку из `/api/profiles` и обновляет ее без
перезагрузки страницы.

## Project Dialogue

Вкладка `Диалог` является основным рабочим экраном проектного разговора. Она
показывает бизнес-действия, а не внутренние названия подсистем: пользователь
выбирает проект и тип задачи, пишет сообщение, прикладывает материалы, получает
ответ и при необходимости сохраняет выводы в память.

Диалог хранит отдельную историю для каждого проекта. Разговор создается
через `POST /api/conversations`, список читается через
`GET /api/conversations?project=<name>`, а новое сообщение добавляется через
`POST /api/conversations/{id}/messages`.

Каждое сообщение собирает контекстный запрос из rolling summary, последних
сообщений и нового текста пользователя, затем проходит обычный Gaia pipeline:
память проекта выбирает контекст, проверка данных маскирует ПД, Forge собирает
безопасный пакет. Этот пакет становится текущим результатом UI и может быть
сразу передан во вкладку `Память` без внешнего скрипта.

История диалога хранится в сервисной зоне `service_docs/Диалоги`, не является
активной памятью Obsidian и не попадает в retrieval сама по себе. Перенос
устойчивых выводов в память выполняется только через вкладку `Память` после
подтверждения выбранных карточек.

Ограничение UX: основной экран `Диалог` не показывает пользователю технические
ярлыки `Lore`, `Veil`, `Scribe`, `LM Studio`, raw prompt или JSON. Эти детали
остаются в разделе `Проверка` и используются для аудита, ручной проверки
источников и подготовки внешнего анализа. Риск утечки ПД остается блокером:
каждый ход разговора проходит обычный pipeline `create_package`, проверку данных
и правила маршрута перед локальным или внешним ответом.

Правая часть шапки показывает один агрегированный бизнес-статус:
`Проверяем готовность`, `Готово к работе` или `Нужна проверка`. Детали про
локальный режим, готовность ответа и проверку данных доступны в подсказке статуса
и в разделе `Проверка`, но не выводятся отдельными индикаторами на главный
экран.

## Focus UX

После завершения задачи обработки Focus показывает результат не как raw JSON, а как рабочий
экран:

- верхний блок `Итог обработки` со статусом, проектом, профилем, внешним
  маршрутом, причиной блокировки и следующим шагом;
- верхняя вкладка `Память` содержит список новых файлов проекта, preview,
  разбор выбранного файла и запись выбранных предложений в память;
- в `Диагностике` остаются вкладки `Итог`, `Проверка данных`,
  `Найденная память`, `Обновление памяти`, `Запрос для модели` и
  `Технические данные`;
- вкладка `Диалог` показывает проектный разговор: список разговоров выбранного
  проекта, историю сообщений, единое поле сообщения/материалов, действия
  `Получить ответ`, `Подготовить материалы`, `Сохранить выводы в память`;
- правая часть шапки содержит один агрегированный статус готовности без
  отдельных технических индикаторов;
- во вкладке `Найденная память` выбранные разделы показаны с checkbox,
  строками, score и matched terms; кнопка `Пересобрать контекст`
  вызывает `POST /api/rebuild-prompt` и обновляет запрос для модели без повторной обработки
  файлов и без повторного маскирования;
- `Evidence` во вкладке `Найденная память` показывает, есть ли confirmed excerpts из
  первичных источников или покрытие осталось partial/missing;
- safe route отображает `Можно во внешний контур после подтверждения`;
- local fallback/PII-risk route отображает `Внешний маршрут заблокирован`;
- полный JSON остается доступным только в технической вкладке.
- пустая форма не отправляет `POST /api/analyze` и показывает
  `Добавь запрос или файл для анализа`;
- если выбран файл без текста запроса, Focus предупреждает, что анализ будет
  строиться по файлу и профилю, но job создается.
- локальный fallback через LM Studio имеет preflight `GET /api/local-status`,
  видимый статус доступности, UI-timeout и кнопку `Остановить ответ`.
- ручные действия обновляют `Итог обработки`: локальный ответ, отмена,
  недоступность LM Studio, обновление памяти, копирование и пересборка контекста показывают
  следующий шаг без необходимости читать raw JSON.
- Обновление памяти неактивно, если пакет требует локальной обработки или есть
  неподтвержденный риск ПД; предложения и запись запускаются только по явному
  клику, а live-память меняется только для выбранных карточек.
- В панели контекста есть управление группами и проектами: создание группы,
  создание проекта, смена названия/статуса/группы выбранного проекта, проверка
  структуры и восстановление недостающих служебных узлов.
- копирование во внешний контур возможно только после безопасного маршрута и
  явной галочки подтверждения.
- пустая форма не создает job ни из UI, ни через прямой API.

## Управление проектами и группами

Gaia хранит источник правды в Obsidian, а не в отдельной базе. Для проекта
создается папка в `Проекты` и минимальный комплект:

- `.gaia-project.json` - код, название, статус, группа и флаг наследования.
- `<код> - Память.md` - входная карта памяти проекта.
- `<код> - Источники.md` - registry источников проекта.
- `<код> - Журнал памяти.md` - журнал обновлений памяти.
- `Память_Graph/<код> - Индекс памяти.md` и папки `00_Core`, `10_Branches`,
  `20_Decisions`, `30_Open_Questions`, `40_Risks`, `50_Sources`, `90_Archive`.

Группы лежат отдельно от `Проекты`, чтобы не попадать в project discovery как
обычные проекты:

```text
Obsidian Vault/Контексты/Группы/<код группы>/
```

Для группы создаются `.gaia-group.json`, `<код> - Контекст.md`,
`<код> - Источники.md`, `<код> - Журнал.md`, `Материалы/Регламенты`,
`Материалы/Шаблоны` и собственный `Память_Graph`. Групповой контекст нужен для
надпроектных регламентов, шаблонов, методик, общих ограничений и правил
качества.

API управления:

- `GET /api/projects` - старый список `projects`, новые `project_records` и
  `groups`.
- `POST /api/projects` - создать проект.
- `PATCH /api/projects/{name}` - обновить код, название, статус, группу и флаг
  наследования; при смене кода Gaia переименовывает файлы с проектным префиксом
  и обновляет markdown-ссылки вида `<старый код> - ...`.
- `POST /api/projects/{name}/validate` - проверить обязательные узлы.
- `POST /api/projects/{name}/repair` - восстановить недостающие служебные файлы
  и папки без перезаписи существующей памяти.
- `POST /api/projects/{name}/archive` - архивировать проект.
- `GET /api/groups`, `POST /api/groups`, `PATCH /api/groups/{code}`,
  `POST /api/groups/{code}/archive` - управление группами.

Формальный контракт endpoints и единый формат JSON-ошибок описаны в
`docs/obsidian_gaia/API.md`.

## UX-9 smoke checklist

- Автопретензии, риск-анализ с файлом и ПД: Veil показывает review, внешний
  маршрут блокируется при unresolved risk, Scribe недоступен.
- Хаб обращений, записка для решения без ПД: пакет готов, запрос для модели
  доступен после review, копирование требует явного подтверждения.
- Blocked ПД-сценарий: `Итог обработки` показывает причину и локальный следующий
  шаг.
- Пустой запрос: Gaia показывает `Добавь запрос или файл для анализа`, job не
  создается.
- Неподдерживаемый файл: API возвращает readable error, JSON остается только в
  технической вкладке.
- LM Studio недоступна: local preflight быстро показывает понятное состояние и
  не выглядит как зависание.
- LM Studio context length: при HTTP 400 вида `tokens to keep ... context length`
  проверь `Context Length` загруженной модели и `local_llm.prompt_char_limit`;
  для 16k context держи лимит Gaia около `16000`, для 8k - `8000-12000`.
- Scribe safe/blocked: safe создает markdown-черновик, blocked недоступен в UI и
  защищен backend `409`.
- Lore section deselect/rebuild: prompt пересобирается без повторного Veil/Echo,
  старый job не меняется.

## Индекс памяти Lore

Lore строит локальный in-process индекс по эффективному контексту:

1. групповой контекст проекта из
   `Obsidian Vault/Контексты/Группы/<код группы>`;
2. префиксная карта `Obsidian Vault/Проекты/*/<код> - Память.md`;
3. активные узлы `Память_Graph`.

Старый `Память.md` поддерживается только как fallback для совместимости.
`90_Archive` исключается из активной индексации. Сервисная документация Gaia в
`Obsidian Vault/Сервисы` или отдельной `service_docs`-папке не индексируется как
проектный контекст. В prompt попадает релевантная выборка разделов памяти, а
`AnalysisPackage.memory_sources` показывает scope `group`/`project`, заголовки,
строки и совпавшие термины.

Выборка строится в два слоя: центральный контекст и контекст запроса.
Групповой слой задает общие правила, а проектный слой имеет приоритет при
конфликте. Ответный слой должен иметь подтвержденный якорь в памяти:
source-summary, ветку, решение, риск или открытый вопрос. Узлы
`Память_Graph/50_Sources` получают приоритет как provenance-якоря, а связанные
wiki-узлы подтягивают релевантные ветки, решения, риски и вопросы. Если по
значимым терминам запроса нет подтвержденного контекста, Lore вставляет блок
`Проверка покрытия Lore` и запрещает достраивать ответ из похожих тем.

`POST /api/rebuild-prompt` принимает `job_id` и
`selected_memory_source_ids`. Endpoint работает только с готовым job и только с
source id, которые уже были выбраны Lore в исходном пакете. Старый job не
меняется; UI получает обновленный prompt и применяет его как текущую рабочую
версию пакета.

Если `lore.query_rewrite = true`, Lore перед deterministic retrieval вызывает
локальную LM Studio только для расширения поисковых терминов. Модель возвращает
массив коротких терминов и не отвечает на пользовательский запрос.

Если `lore.semantic_rerank = true`, Lore после deterministic retrieval отправляет
в локальную LM Studio только карточки уже найденных candidate-секций. Модель
может переупорядочить существующие `id`, но не может добавить новый источник,
новый факт или новый раздел. При неизвестном `id`, плохом JSON, пустом выборе,
недоступности LM Studio или выборе без прямого якоря по теме Gaia отбрасывает
ИИ-rerank и использует обычный deterministic Lore.

Если `lore.gap_detector = true`, Lore после выбора источников просит локальную
модель оценить покрытие запроса статусом `ok`, `partial`, `weak` или `none`.
Эта диагностика добавляется в effective context как служебный блок и не является
новым источником фактов.

Если `lore.veil_llm_review = true`, Veil после rule-based маскирования запускает
локальную LLM-проверку остаточного риска ПД. LLM review может только усилить
риск и заблокировать внешний маршрут; он не отменяет находки правил.

Если `lore.scribe_candidate_classifier = true`, Scribe использует локальную
классификацию кандидатов: решения, правила, риски, открытые вопросы,
технические факты и элементы, которые не стоит переносить в память. LLM остается
assist-layer: apply разрешен только через staged plan, safety checks и review.

Если `lore.project_health_llm = true`, `POST /api/projects/{name}/validate`
возвращает расширенную диагностику здоровья памяти по структурной сводке проекта.
Диагностика не чинит проект сама и не удаляет источники.

Hearth перед отправкой полного локального ответа сокращает prompt до
`local_llm.prompt_char_limit` и ограничивает генерацию через `local_llm.max_tokens`.
Эти настройки должны соответствовать фактическому `Context Length`, с которым
модель загружена в LM Studio, а не только заявленному окну модели.

Финальный локальный ответ Hearth запрашивается как структурированный JSON:
краткий вывод, ключевые наблюдения, риски, открытые вопросы и следующие шаги.
Focus отображает такой ответ карточками. Если локальная модель вернула обычный
текст или невалидный JSON, Gaia использует прежний текстовый fallback и не делает
повторный LLM-вызов. Такой же карточный рендер используется для локальных ответов
в Project Dialogues. Служебные LLM-вызовы Lore rerank, gap detector, Veil review,
Scribe classifier и project health остаются узкими assist-layer вызовами и не
переводятся в пользовательские карточки.

## Archive retention

Archive разделяет рабочие артефакты и audit безопасности:

- `runs/` - временные рабочие папки запусков.
- `Сервисы/Gaia Local Analytics/Журнал запросов` - рабочие журналы с prompt.
- `Сервисы/Gaia Local Analytics/Аудит безопасности` - короткий audit без полного
  prompt, текста вложений и проектной памяти.

Очистка запускается вручную:

```bash
python3 -B -m gaia.archive --cleanup
```

Безопасная проверка без удаления:

```bash
python3 -B -m gaia.archive --dry-run
```

Retention никогда не чистит `Obsidian Vault/Проекты`.

## Финальная проверка

Проверено: локальный API, профили, Veil, Lore, Forge, Scribe, Archive retention,
локальный Echo smoke и blocked local fallback. Внешние вызовы не выполнялись.

Канонические проверки после изменения кода:

```bash
python3 -B -m unittest discover -s tests
python3 -B -m gaia.config
```

## Документация сервиса

```text
<service_docs>/Gaia Local Analytics
```

Это техническая документация для развития сервиса, а не пространство проектной
памяти. Она не содержит проектную префиксную карту памяти и не должна
индексироваться как проектный контекст.
