# Отчёт об исправлении запуска и отдельного окна Gaia

## 1. Исходная проблема

Диагностика в [RUNTIME_DIAGNOSTIC.md](RUNTIME_DIAGNOSTIC.md) зафиксировала осиротевший `osascript`/WKWebView: окно существовало, но сервер на его loopback-порту отсутствовал. Исходный продуктовый коммит — `c37eea9`.

## 2. Первичная причина

`app.py` запускал сервер и сразу открывал JXA-окно. Процесс `osascript` не был связан с жизненным циклом сервера: после остановки сервера WKWebView мог сохранять загруженный DOM, а launcher не проверял, что этот DOM относится к живому конкретному запуску.

## 3. Реализованный жизненный цикл

`launcher → server start → GET /api/runtime → runtime fingerprint → window start с runtime в URL → frontend health monitoring → безопасный экран при потере сервера или смене runtime → завершение дочернего окна при остановке сервера`.

`gaia.server.main(open_window=True)` сначала запускает HTTP-обслуживание в служебном потоке. Затем `launch_gaia_window` ожидает ответ Gaia, а не только открытый TCP-порт. При неготовности или занятом порту окно не создаётся.

## 4. Runtime fingerprint

`GET /api/runtime` выдаёт только: `ready`, `runtime_id`, `pid`, `started_at`, `git_commit`, `api_contract_version`, `frontend_hash`.

Идентификатор создаётся один раз при импорте `gaia.runtime` и неизменен до завершения процесса. Окно получает его в query-параметре `runtime`, а frontend сравнивает с периодическим ответом `/api/runtime`. В fingerprint намеренно нет материалов, проектов, текстов, находок, псевдонимов, токенов, конфигурации модели и путей пользовательских данных.

Пример безопасной формы: `{"ready":true,"runtime_id":"…","pid":123,"git_commit":"…"}`.

## 5. Изменённые файлы

| Файл | Изменение |
| --- | --- |
| `gaia/runtime.py` | Создаёт и отдаёт безопасный fingerprint запуска. |
| `gaia/server.py` | Добавляет `/api/runtime`, ожидает готовности перед окном, закрывает дочернее окно при остановке. |
| `gaia/launchers.py` | Проверяет runtime через loopback, передаёт runtime в URL, предотвращает повторное окно в одном процессе. |
| `gaia/static/index.html` | Блокирует прежний DOM экраном потери связи при недоступности или смене runtime. |
| `tests/test_launchers.py`, `tests/test_runtime_lifecycle.py` | Покрывают готовность, занятый чужой порт, повторный запуск и runtime-binding. |
| `docs/USER_INTERFACE.md` | Описывает понятное поведение при потере связи. |

## 6. Контракты

- Готовность: `GET /api/runtime` с `ready:true` и ожидаемым `runtime_id` в пределах восьми секунд.
- Тайм-аут: окно не открывается; launcher сообщает, что Gaia не подтвердила готовность.
- Потеря связи: frontend заменяет содержимое безопасным экраном и блокирует действия.
- Новый runtime: старое окно видит несовпадение идентификатора и блокируется.
- Занятый порт: неизвестный процесс не завершается; сервер не запускается и окно не открывается.
- Повторная команда в том же серверном процессе не создаёт второе окно.
- Штатное и аварийное завершение `main`: зарегистрированное дочернее окно получает `terminate`.

## 9. Отдельное окно

JXA-окно запускалось фактическим штатным способом только после готовности runtime. Системный `FilePanelDelegate` сохранил регистрацию через `$.GaiaFilePanelDelegate.alloc.init`; drag-and-drop и браузерный `<input type=file>` не менялись. Клик по нативной панели выбора файла не автоматизировался и остаётся ручной проверкой.

## 10. Браузер

Browser-версия продолжает работать без runtime-параметра. После обновления страницы она получает текущий HTML и новый fingerprint; `Cache-Control: no-store` сохранён. Окно с runtime-параметром не может незаметно присоединиться к другому запуску.

## 11. Полная проверка

Выполнены: целевые launcher/runtime-тесты, полный `unittest discover`, Python `compileall` с временным bytecode-cache, `node --check gaia/gaia_window.js` и `git diff --check`.

## 12. Git

HEAD до исправления: `ddc2fc8`. Новый коммит указан после завершения проверки. В него не включаются `config.json`, storage, exports, PID/логи, временная конфигурация и `.agents/`. Push не выполняется.

## 13. Ограничения

Автоматизированы runtime-сигнал, блокировка DOM и завершение дочернего окна при штатном завершении сервера. Нельзя надёжно автоматически нажать системную файловую панель macOS в этом контуре. Аварийное убийство процесса извне покрывается frontend-проверкой при следующем интервале, но визуальный переход необходимо один раз подтвердить вручную.

## 14. Инструкция ручной приёмки

1. Закрыть старые окна Gaia.
2. Убедиться, что `127.0.0.1:8787` свободен.
3. Запустить Gaia ярлыком.
4. Открыть `http://127.0.0.1:8787/api/runtime` и записать runtime_id.
5. Остановить сервер через `Ctrl+C` в окне launcher.
6. Убедиться, что окно закрылось либо показало экран потери связи.
7. Запустить Gaia снова и убедиться, что runtime_id новый.
8. Начать синтетическую ручную приёмку этапа 6.

## 15. Итог

Жизненный цикл запуска и отдельного окна исправлен и готов к ручной приёмке.
