# План укрепления и оптимизации Gaia

Статус: GitNexus настроен; реализация оптимизаций не начата.

Дата аудита: 2026-07-11.

## Границы работ

- GitNexus устанавливается как локальный инструмент анализа зависимостей и радиуса изменений.
- Индекс строится без embeddings и без генерации wiki через внешнюю модель.
- Код, конфигурация и поведение Gaia меняются только после отдельного подтверждения пользователя.
- Перед каждым этапом требуется локальный backup, после этапа — обновление документации и полный прогон тестов.
- ПД и содержимое проектной памяти не должны передаваться внешним сервисам.

## Этап 0. GitNexus

Цель: получить локальный граф импортов, вызовов, API-маршрутов и затрагиваемых тестов.

Статус на 2026-07-11: выполнено.

- Установлен GitNexus `1.6.9` в пользовательский `PNPM_HOME`.
- Репозиторий `Gaia-LAS` проиндексирован: 1 609 узлов, 3 512 связей,
  50 кластеров и 140 execution flows.
- Индекс соответствует commit `3ebbe38` ветки `main`.
- Embeddings не создавались; анализ выполнен с `--drop-embeddings`.
- MCP зарегистрирован только для Codex через абсолютный путь к локальному
  executable.
- Установлены стандартные GitNexus skills; в репозитории созданы `AGENTS.md`,
  `CLAUDE.md` и `.claude/skills/gitnexus/`.
- Локальная база `.gitnexus/` исключена из Git через `.git/info/exclude`.
- FTS extension в данном окружении недоступен; GitNexus продолжает работать без
  FTS-функций. Граф, clusters и execution flows построены успешно.
- Для появления MCP-инструментов в интерфейсе требуется открыть новую сессию
  Codex или перезапустить приложение: текущая сессия не перечитывает MCP config.

Порядок настройки:

1. Установить официальный пакет `gitnexus` локально.
2. Выполнить индексирование репозитория без `--embeddings`; для гарантированного
   удаления ранее созданных векторов использовать `--drop-embeddings`.
3. Подключить MCP только для Codex.
4. Проверить статус индекса, список процессов и доступность impact-analysis.
5. Не запускать `gitnexus wiki` без явно настроенной локальной модели.

Критерии готовности:

- репозиторий виден GitNexus — выполнено;
- индекс актуален — выполнено;
- Codex получает инструменты `context`, `impact`, `trace`, `route_map` и
  `detect_changes` — конфигурация выполнена, активация проверяется в новой сессии;
- в настройках нет облачного LLM или внешней отправки исходного кода — выполнено.

## Этап 1. Жесткая гарантия локальности

Приоритет: критический.

Статус на 2026-07-12: выполнено.

Планируемые изменения:

- LLM endpoint проходит проверку scheme и literal loopback IP;
- внешний provider URL блокируется для всех маршрутов Lore, Veil, Scribe,
  Hearth и Project Health;
- добавлены негативные config-тесты и проверки IPv4/IPv6 loopback;
- provider name не считается доказательством локальности.

Критерий приемки: конфигурация с нелокальным endpoint не запускает чувствительный маршрут и возвращает понятную ошибку.

## Этап 2. Реальные таймауты и отмена задач

Приоритет: критический.

Статус на 2026-07-12: выполнено.

Планируемые изменения:

- добавлены отдельные configurable timeout для job и транскрибатора;
- активный процесс транскрибации завершается при отмене и timeout;
- job получает отдельный terminal status `cancelled`, не маскирующий отмену как
  успешный или обычный failed result;
- добавлен локальный `POST /api/jobs/<job_id>/cancel` и кнопка Focus
  «Остановить подготовку»;
- добавлены тесты отмены worker и остановки процесса транскрибации.

Критерий приемки: зависшая операция освобождает worker и не продолжает скрытые записи после terminal status.

## Этап 3. Retention и минимизация чувствительных данных

Приоритет: высокий.

Статус на 2026-07-12: механизм выполнен; фактическое удаление накопленных
runs ожидает отдельного подтверждения.

Планируемые изменения:

- добавлен обезличенный retention-status в `service_docs/Archive` и
  `GET /api/retention-status`;
- завершенные jobs и их полный package очищаются из памяти через 1800 секунд;
- runs, исходные uploads и journals остаются под существующей retention policy;
- dry-run выявил накопленные runs старше 7 дней, поэтому
  `retention.cleanup_on_startup` намеренно оставлен `false` до отдельного
  разрешения на их физическое удаление.

Критерий приемки: исходные файлы и полные prompts не остаются бессрочно без видимого предупреждения.

## Этап 4. Атомарность и конкурентная запись

Приоритет: высокий.

Статус на 2026-07-12: выполнено.

Планируемые изменения:

- добавлен единый helper атомарной записи через временный файл и `replace`;
- добавлены локальные блокировки для диалогов, metadata проектов/групп и
  Scribe Inbox state;
- conversation write, Project Registry, Inbox state и Scribe bookkeeping
  переведены на атомарное сохранение;
- Scribe node updates и замена ссылок при смене project code также используют
  атомарную запись.

Критерий приемки: параллельные запросы не повреждают JSON и не теряют подтвержденные изменения.

## Этап 5. Защита локального HTTP API

Приоритет: высокий.

Статус на 2026-07-12: выполнено.

Планируемые изменения:

- config запрещает non-loopback bind;
- сервер выдает HttpOnly SameSite локальный session cookie;
- каждый `POST` и `PATCH` проверяет loopback client, cookie и совпадающий
  `Origin`;
- при нарушении возвращается стабильный `403 mutation_not_authorized`;
- защита распространяется на Project Registry, Scribe apply, jobs, диалоги,
  launch и все другие изменяющие endpoints.

Критерий приемки: сторонняя веб-страница и неавторизованный локальный запрос не могут изменить проектную память.

## Этап 6. Контроль памяти и очереди

Приоритет: средний.

Статус на 2026-07-12: выполнено.

Дополнение на 2026-07-12: HTTP-разбор multipart переведён на потоковое чтение;
контрактный тест проверяет, что чтение не превышает 64 KiB за операцию.

Планируемые изменения:

- ограничен multipart body до 25 MB, один файл до 20 MB и количество файлов
  до восьми;
- параллельный разбор uploads ограничен двумя слотами;
- очередь анализа ограничена четырьмя активными и восемью ожидающими jobs;
- перегрузка получает контролируемый `429 job_queue_full` или
  `503 upload_capacity_full`;
- multipart body разбирается потоково блоками по 64 KiB: сервер больше не
  держит одновременно полное HTTP-body и дерево MIME. Для совместимости с
  текущими jobs содержимое каждого принятого файла пока хранится в памяти
  один раз; переход к zero-copy передаче файлов между HTTP и worker остаётся
  отдельной архитектурной задачей.

Критерий приемки: параллельные загрузки не создают неконтролируемый рост RAM и не блокируют сервис без явного ответа.

## Этап 7. Управляемая декомпозиция

Приоритет: средний, после стабилизации runtime.

Статус на 2026-07-12: выполнено для UI; доменные границы `memory.py` и
`scribe.py` сохранены плоскими по принятому проектному соглашению.

Планируемые изменения:

- HTML, CSS и JavaScript вынесены в `gaia/static/index.html`;
- `gaia.ui` сохранен как совместимый тонкий загрузчик `INDEX_HTML`, поэтому
  HTTP-контракт, UI-тесты и визуальное поведение не изменены;
- `scribe.py` и `memory.py` не разнесены на подпакеты: их существующие
  функции уже разделяют планирование, валидацию, применение и retrieval, а
  дополнительная механическая нарезка не дает функциональной пользы;
- перед изменением были выполнены GitNexus impact-analysis и проверка flows.

Критерий приемки: поведение UI и API не меняется, а крупные файлы перестают быть единственной точкой изменений.

## Порядок реализации

Выполнены критические этапы 1 и 2, высокие этапы 3–5 и средние этапы 6–7.

Каждый этап оформляется отдельным изменением:

1. backup затрагиваемых файлов;
2. GitNexus impact report;
3. реализация только одного этапа;
4. целевые тесты;
5. полный `python3 -B -m unittest discover -s tests`;
6. `python3 -B -m gaia.config`;
7. проверка `git diff --check`;
8. обновление документации;
9. ручное подтверждение перед commit или push.

## Что не входит в текущую работу

- изменение кода Gaia;
- включение автоматической очистки;
- изменение providers или маршрутов моделей;
- рефакторинг UI, Scribe или Lore;
- commit и push;
- генерация GitNexus wiki через внешнюю модель.
