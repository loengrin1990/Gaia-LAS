# Scribe

## Роль

Scribe готовит и применяет обновления проектной памяти внутри Gaia. В UI он
называется `Scribe: обновление памяти`: имя подсистемы остается видимым, а
кнопки описывают бизнес-действия. Он заменяет ручную цепочку
`черновик -> внешний skill -> ручное распределение` встроенным workflow, но не
пишет live-память без явного review.

## Уровни работы

1. `Scribe Inbox` находит файлы внутри выбранного проекта и превращает их в
   обычный безопасный `AnalysisPackage`.
2. `Scribe draft` создает markdown-черновик в сервисной документации. Это
   fallback/export-режим; проектная память не меняется.
3. `Scribe plan` строит структурированные предложения из готового job,
   диалогового turn или Inbox package:
   карточки кандидатов, operation, destination, confidence, evidence и preview.
   В UI это действие называется `Предложить записи в память`.
4. `Scribe apply` применяет только выбранные пользователем карточки.
   В UI это действие называется `Записать выбранное в память`.

## Quality Gate

Перед `apply` Scribe пропускает план через deterministic quality gate. Это
обязательный слой, который сильнее локальной LLM-классификации:

- все active memory-узлы, включая `50_Sources`, получают короткое имя после
  проектного префикса: максимум 3 смысловых токена; токены считаются по
  пробелам, дефисам и underscore;
- короткое имя все равно должно быть смысловым: hash, числовой id, digest или
  служебный slug не проходят Quality Gate как memory-title;
- если имя файла низкосигнальное, Scribe строит memory-title из первой
  смысловой строки извлеченного текста и создает content-based source-summary;
- реальные файлы в `Исходники/` и `Транскрипции/` не переименовываются, длинные
  имена остаются в `source`, `Evidence` и `Provenance`;
- `source_summary` не наследует имя файла: он получает memory-name вроде
  `Шаги запуска`, `Паспорт OCR`, `Защита ПДн`, `Баги бэклог`,
  `Роли MVP2`, `Сценарии MVP2`, `Модель данных`, `Статусная модель`;
- если несколько материалов описывают один контекст, Scribe делает один
  `merge`-item с несколькими provenance, а не несколько похожих active nodes;
- карточки без reusable context, generic fallback и похожие на raw dump
  получают `skip/exclude` и не могут быть применены как память;
- active node не должен создаваться с `links: []`: render добавляет связь с
  graph index, а index/journal/sources обновляются append-only.

Scribe plan теперь трактуется как набор операций, а не просто список markdown
файлов: `create`, `merge`, `archive`, `skip`. MVP автоматически применяет
безопасные `create/merge` для новых узлов; рискованные обновления существующих
решений остаются задачей review.

## Semantic enrichment

Scribe отвечает не только за регистрацию источников. Его рабочий контракт:
исходники остаются исходниками, а в память попадает структурированная модель
проекта. Поэтому `source_summary` в `50_Sources` хранит provenance: имя файла,
тип, путь, статус извлечения и факт, что сырой текст не переносился. Сырой
transcript, длинные цитаты и табличные dumps не должны становиться body
memory-node.

`source_summary` создается только из реально выбранного файла/Inbox item как
provenance-узел с коротким memory-name. Scribe не создает `source_summary` из
Lore `confirmed excerpt`: такие excerpts уже являются контекстом для анализа,
но не являются готовыми узлами памяти и не должны попадать в graph как
обрезанный dump.

Смысловые узлы создаются отдельно:

- архитектура, процессы, интеграции и роли систем -> `10_Branches`;
- принятые решения и правила -> `20_Decisions`;
- открытые вопросы и развилки -> `30_Open_Questions`;
- риски качества данных, процесса и приемки -> `40_Risks`.

Для длинных встреч Scribe строит enrichment-кандидаты из безопасного
`masked_text`: локальный LLM-классификатор может добавить карточки, но
deterministic semantic pass тоже обязан извлечь устойчивые сигналы из
материалов. Такие карточки пишутся как будущие markdown-узлы с секциями:
`Суть`, `Контекст`, `Как использовать в Gaia`, `Evidence`. Они должны
объяснять, что именно известно и как использовать это в будущих запросах, а не
формулировать "обсуждалось X".

Если deterministic pass видит только общие слова вроде "архитектура",
"процесс", "система" или "отчет", но не находит доменный паттерн, карточка
уходит в `exclude` и не может быть применена как память. Generic fallback без
предметного контекста считается сигналом к ручному enrichment, а не memory-node.

Acceptance-критерий для проекта вроде `ГДРС-отчет`: после обработки встреч
пользователь должен получить узлы о текущем потоке данных, ролях СКУД,
синхронизатора, Face ID/отчетности, ботах, разовых пропусках, ручном журнале
охраны, мастер-системе названий и рисках расхождения данных. Gaia должна
находить эти узлы через Lore без повторного чтения transcript.

## Inbox

Inbox переносит обновление памяти в интерфейс Gaia для файлов, которые уже
лежат в проектном пространстве. Он сканирует проект, исключает служебную память,
graph nodes, `Автоизвлеченный текст` и уже записанные источники, показывает
только файлы, которые еще требуют обработки, затем по кнопке `Разобрать
выбранный файл` готовит безопасный контекст через тот же pipeline, что и
обычный анализ: extraction, Veil, Lore, Forge.

В UI Scribe находится в верхней вкладке `Память`, а не внутри `Диагностики`.
`Диагностика` остается местом для технических данных, запроса для модели, Veil
и Lore-деталей.

Список файлов - это список кандидатов на обработку. Scribe не читает все эти
файлы в память автоматически. Реально разбирается только выбранный файл после
действия `Разобрать выбранный файл`; live-память меняется только после
`Предложить записи в память` и `Записать выбранное в память`.

Для Inbox package предложения Scribe ограничены выбранным файлом. Он не добавляет
`source_summary` из Lore/evidence других документов проекта, даже если они
попали в контекст package.

Статусы Inbox хранятся в `service_docs/Scribe Inbox State`. `prepared` означает,
что файл уже разбирали; `indexed` означает, что выбранные записи уже применены
в память и файл скрывается из списка; `ignored` скрывает файл из рабочего
потока пользователя, но не удаляет исходник.

Inbox не пишет память сам. После разбора файла пользователь просит Scribe
предложить записи, проверяет карточки и только потом запускает запись выбранного
в память.

## Excel

Для `.xlsx` Gaia использует структурную нормализацию без зависимости от
`openpyxl`: читает workbook XML, shared strings, листы, заголовки, первые
строки, merged ranges и ячейки с формулами. Локальная LLM и Scribe получают не
сырой Excel-файл, а markdown preview.

Цель preview - помочь выделить устойчивые выводы: source-summary, правила,
статусы, риски и открытые вопросы. Сырые строки таблицы не должны переноситься
в память целиком.

## Safety contract

- Scribe блокируется при unresolved PII.
- UI также блокирует Scribe, если пакет требует local fallback или ручной
  проверки.
- Локальная LLM может только улучшить классификацию кандидатов; deterministic
  rules и safety layer остаются обязательными.
- LLM не может записывать память, снимать PII-блокировку, перезаписывать
  решения или создавать факты без evidence.
- Перед apply создается backup активной памяти в `service_docs/Scribe Backups`.

## Routing

Scribe распределяет кандидаты так:

- `decisions`, `rules` -> `Память_Graph/20_Decisions`;
- `risks` -> `Память_Graph/40_Risks`;
- `open_questions` -> `Память_Graph/30_Open_Questions`;
- `technical_facts` -> `Память_Graph/10_Branches`;
- confirmed evidence -> `Память_Graph/50_Sources`;
- `exclude` -> не применяется.

На MVP apply создает новые graph nodes и append-only записи в index, sources и
journal. Рискованные merge/update существующих decisions остаются review-задачей,
а не автоматическим действием.

## API

- `GET /api/scribe-inbox?project=<name>` - список еще не записанных файлов-кандидатов проекта.
- `GET /api/scribe-inbox/preview?project=<name>&path=<relative>` - preview
  файла; для `.xlsx` возвращает структурный Excel preview.
- `POST /api/scribe-inbox/package` - разобрать выбранный файл.
- `POST /api/scribe-inbox/ignore` - скрыть файл из списка.
- `POST /api/scribe-draft` - сохранить markdown-черновик.
- `POST /api/scribe-plan` - построить staged plan для готового job.
- `POST /api/scribe-apply` - применить выбранные `selected_item_ids`.
