# Project Dialogues

Project Dialogues превращает вкладку `Диалог` в основной проектный разговор.
Каждый проект получает собственный список разговоров, а каждое новое сообщение
использует rolling summary и последние сообщения как часть контекстного запроса
для Lore/Forge.

Основной экран строится на бизнес-действиях:

- `Получить ответ` - проверить данные, собрать контекст и ответить локально;
- `Подготовить материалы` - собрать безопасный пакет без немедленного ответа;
- `Сохранить выводы в память` - передать текущий результат в staged review
  памяти.

В пользовательском экране не показываются внутренние ярлыки `Lore`, `Veil`,
`Scribe`, `LM Studio`, raw prompt или JSON. Они остаются в разделе `Проверка`,
где нужны для аудита, источников и ручного review.

Шапка Gaia также не показывает отдельные технические индикаторы готовности.
Справа остается один бизнес-статус:

- `Проверяем готовность` - Gaia еще проверяет доступность локального ответа или
  находится в осторожном локальном режиме;
- `Готово к работе` - локальный ответ доступен, а проверка данных не нашла
  блокеров;
- `Нужна проверка` - ответ недоступен или проверка данных требует внимания.

Подробности хранятся в подсказке статуса и в разделе `Проверка`.

## Хранение

Диалоги лежат в `service_docs/Диалоги/<project-slug>/<conversation-id>.json`.
Это сервисная зона Gaia, а не активная Obsidian-память проекта. Поэтому история
диалога не попадает в retrieval автоматически и не меняет `Память_Graph` без
явной записи через Scribe.

Сущности:

- `Conversation` - id, проект, title, status, created/updated, rolling summary,
  messages и последний job/package id;
- `ConversationMessage` - role, исходный текст, masked text, время, route,
  safety status и связанный run id.

Архивирование меняет `status = archived`; такие диалоги не показываются в
списке проекта, но файл остается в сервисной истории.

## Контекстное сообщение

При отправке нового сообщения Gaia:

1. читает выбранный диалог;
2. собирает contextual query из rolling summary, последних сообщений и нового
   сообщения пользователя;
3. запускает обычный `create_package` с выбранным проектом и профилем;
4. сохраняет user message с masked text и safety status;
5. опционально отправляет prompt в LM Studio и сохраняет assistant message;
6. обновляет rolling summary и возвращает свежий `AnalysisPackage`.

Rolling summary является локальной сжатой историей, а не источником истины. Для
ответов с высокой важностью Lore по-прежнему должен подтверждать факты через
проектную память и evidence из источников.

## UI

Во вкладке `Диалог` Focus показывает:

- список активных разговоров выбранного проекта;
- историю сообщений выбранного диалога;
- единое поле сообщения и материалов;
- кнопки `Новый разговор`, `Получить ответ`, `Подготовить материалы`,
  `Сохранить выводы в память`, `Архивировать`.

После отправки сообщения свежий package становится текущим результатом
интерфейса: обновляются summary, вкладки Диагностики и доступность Scribe.
Если пользователь нажал `Получить ответ`, локальный результат сохраняется как
единственная карточка assistant в истории разговора. Для только что полученного
результата прямо в этой карточке доступны действия `Уточнить запрос`,
`Сохранить выводы в память` и `Показать подготовленный запрос`; второй
дублирующий блок ответа не выводится. Пока Gaia ожидает ответ, над состоянием
диалога отображается заметный индикатор работы без искусственного процента.
Если prompt был сокращен под окно локальной модели, Gaia сообщает об этом в
статусе; полная версия prompt остается в разделе `Проверка`.

Кнопка `Сохранить выводы в память` в нижней панели скрывается, когда это же
действие доступно у текущего ответа. Для пакета, подготовленного без немедленного
ответа, нижняя кнопка остается точкой входа в staged review. Название разговора
в списке сокращается визуально, но полная безопасно замаскированная формулировка
доступна в подсказке при наведении и в доступном имени кнопки.

Риск утечки ПД является блокером. Запрещено превращать Project Dialogues в
прямую отправку истории разговора в модель: каждый ход обязан проходить обычный
pipeline `create_package`, проверку данных, правила маршрута и только затем
локальный ответ или подготовку внешнего анализа.

## Связь со Scribe

Scribe endpoints принимают не только `job_id`, но и готовый `package` из
диалогового сообщения. Это позволяет построить предложения и применить выбранные
карточки памяти прямо из результата разговора, без внешних скриптов.

Scribe по-прежнему соблюдает ограничения:

- не работает при unresolved PII;
- не применяет изменения без review выбранных карточек;
- создает backup активной памяти перед записью;
- не переносит всю историю диалога в память автоматически.

## API

- `GET /api/conversations?project=<name>` - активные диалоги проекта.
- `GET /api/conversations/<conversation_id>` - один диалог.
- `POST /api/conversations` - создать диалог.
- `POST /api/conversations/<conversation_id>/messages` - добавить сообщение.
- `POST /api/conversations/<conversation_id>/archive` - архивировать диалог.

## Границы текущего этапа

Реализовано: storage, API, UI-история per project, продолжение существующего
диалога, rolling summary/context window и связь с предложениями/записью Scribe.

Отложено на полировку: rename, поиск по истории, export и восстановление
архивированных диалогов из UI.
