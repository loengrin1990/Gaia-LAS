# Lore

## Роль

Lore выбирает релевантные разделы эффективного контекста для аналитического пакета Gaia. Код находится в `Local_Analytics_System/gaia/memory.py`, а управление проектами и группами вынесено в `gaia/projects.py`.

Эффективный контекст состоит из двух уровней:

1. групповой контекст проекта из `Obsidian Vault/Контексты/Группы/<код группы>`;
2. проектная память из `Obsidian Vault/Проекты/<project>/<код> - Память.md` и активных узлов `Память_Graph`.

Старый `Память.md` поддерживается как fallback для совместимости.

## Граница индекса

Lore не индексирует сервисную документацию Gaia и не считает группы отдельными проектами. Группы живут вне `Проекты`, а `project_dirs()` возвращает только проектные папки с memory-файлом.

`90_Archive` исключается из активной индексации и на уровне проекта, и на уровне группы.

## MemorySource

`MemorySource` теперь содержит `scope`:

- `group` - источник пришел из надпроектной группы;
- `project` - источник пришел из проектной памяти.

Эти данные попадают в `AnalysisPackage.memory_sources`, prompt Forge, Archive-журнал и Focus UI.

## Evidence plan

После выбора `MemorySource` Lore строит доказательный слой `evidence_plan`.
Это не замена проектной памяти и не миграция vault: память остается компактной
картой смысла, а первоисточники подключаются точечно.

Порядок evidence-layer:

1. если среди выбранных секций уже есть первичный файл из `Исходники` или
   `Транскрипции`, Lore добавляет короткий `confirmed` excerpt;
2. если выбран source-summary или graph-узел, Lore делает drill-down по
   первичным файлам проекта и ищет 1-3 коротких подтверждающих excerpt;
3. если первичный файл не найден, Lore оставляет `partial` evidence от
   source-summary или выбранной памяти;
4. если разделы памяти не выбраны, evidence получает `missing`.

`EvidenceItem` хранит claim, status, source id/path, heading, excerpt, reason и
scope. Forge добавляет блок `# Evidence plan Lore` в prompt, Archive пишет
краткую сводку в журнал и safety audit, Focus показывает summary в Inspector.
Ответ должен использовать `confirmed` как подтверждение, а `partial/missing`
явно трактовать как неполное покрытие.

## Контракт выбора

Центральный слой дает рамку контекста. Ответный слой должен иметь подтвержденный якорь в памяти: source-summary, ветку, решение, риск или открытый вопрос. Узлы `Память_Graph/50_Sources` получают приоритет как provenance-якоря, а связанные wiki-узлы подтягивают релевантные ветки, решения, риски и вопросы.

Групповой контекст задает общие регламенты, шаблоны, методики и ограничения. При конфликте проектная память имеет приоритет, а ответ должен явно назвать конфликт.

Если по значимым терминам запроса нет подтвержденного контекста, Lore вставляет блок `Проверка покрытия Lore` и запрещает достраивать ответ из похожих тем.

Evidence plan усиливает этот контракт: он не разрешает модели придумывать
факты, а помогает отличить выбранную память от первично подтвержденных
фрагментов.

## Локальные ИИ-слои Lore

Если в `config.json` включен `lore.query_rewrite`, Lore использует route `lore_query_rewrite` до retrieval для расширения запроса поисковыми терминами. Модель возвращает только массив `terms`; она не отвечает на пользовательский запрос, не добавляет факты и не создает источники.

Если включен `lore.semantic_rerank`, Lore использует route `lore_rerank` как второй слой ранжирования. Deterministic retrieval остается первым и обязательным: модель получает только карточки уже найденных candidate-секций и может вернуть только их `id`.

Если включен `lore.gap_detector`, Lore после выбора источников просит route `lore_gap_detector` оценить покрытие запроса. Допустимые статусы: `ok`, `partial`, `weak`, `none`. Диагностика добавляется как служебный блок и не является источником фактов.

Антигаллюцинационные ограничения:

- неизвестный `id` от модели инвалидирует весь rerank;
- плохой JSON, пустой выбор или недоступность provider дают fallback к deterministic Lore;
- модель не может добавить новый источник, текст, факт или заголовок;
- если выбранные моделью секции не содержат прямого focus-якоря запроса, rerank отбрасывается;
- блок `Проверка покрытия Lore` создается только deterministic-слоем, а не моделью.
- query rewrite влияет только на токены поиска;
- gap detector оценивает только выбранные sources и не может менять выборку.

## Другие локальные LLM-усилители

Veil может запускать route `veil_review` после rule-based маскирования. Локальная модель проверяет только остаточный риск ПД в уже обработанном тексте и может только усилить блокировку внешнего маршрута.

Scribe может запускать route `scribe_classifier` при создании черновика памяти. Классификация добавляет раздел с кандидатами, но не применяет изменения к проектной памяти.

Project Registry может запускать route `project_health` при validate. Модель получает только структурную сводку проекта: количество источников, graph-узлов, registry-статусов и текущих issues. Она возвращает диагностику, но не выполняет repair и не меняет файлы.

## Пересборка prompt

`POST /api/rebuild-prompt` принимает только `MemorySource.id`, выбранные в исходном job. Rebuild перечитывает и групповые, и проектные узлы, поэтому выбранные групповые регламенты не теряются при ручной пересборке.

При rebuild evidence plan пересчитывается по оставшимся выбранным секциям.
Новый prompt содержит обновленный блок `Evidence plan Lore`.
